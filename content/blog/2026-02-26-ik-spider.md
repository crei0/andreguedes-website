# What are FK (Forward Kinematics) and IK (Inverse Kinematics)

As some context, the differences between FK and IK are __mainly__ the direction of which bone __controls__ the others...

For example in a human arm, you might have something like a bone hierarchy that goes from the spine to the wrist.

![Example using Godot's SkeletonProfileHumanoid](/img/2026-02-26-ik-spider/fk-vs-ik-human bones.jpg)

Simplifying the explanation here, but when using `FK` the direction of which bone __controls__ the others would be from the **spine to the wrist** (being forward to the end bone, the wrist). so if you rotate the shoulder bone, then all the __children__ (until the wrist) would also rotate with it.

With `IK` the direction is the inverted, going from the **wrist to the spine**, so if the wrist bone moves away from the spine then the entire hierarchy will be __pulled__ with it. This is useful because we want each __foot bone__ of the spider to be __in control__ forcing the other __child__ bones to change accordingly.

For a visual explanation see [How 3D Animation Works: FK vs. IK](https://www.youtube.com/shorts/wCPYtaVuW2w)

## Making the 3D spider in Blender and figuring out an export to Godot workflow

Initially I tried to make the entire 3D model (with all the limbs) and `Armature` (bone hierarchy) as a single object in Blender and export it to Godot as `.GLB`, but if I need to do changes to the Blender hierarchy it caused problems, so I split the Spider body and one Spider limb into separate Blender `collections`.

![Blender's hierarchy](/img/2026-02-26-ik-spider/spider--blender-hierarchy.jpg)

Keeping the 2 parts of the Spider as separate Blender `collections` allowed me to modify and export only the part I changed, this __reduced__ the number of problems after importing the files into Godot's scenes, because before I had my `IK node hierarchy` inside the `Skeleton3D` and if I renamed objects/bones/etc on Blender it could break the reference inside Godot's Scene, because the `3D mesh` and the `Skeleton3D` would lose the reference.

### How does the `body` and `limbs` relation works

The `body` is the one that moves using the user inputs.

The `limbs ` are children of the body, so they to move with it... but because we want the __feet__ __nodes__ to move independently of the body we code their position and the IK nodes solve the remaining leg/limb bones rotation and position,

#### The `FABRIK3D` node

Inside the __imported__ `Skeleton3D` there's the `Fabrik3D` and it's a **F**orw**a**rd and **F**orward and **B**ackward **R**eaching **I**nverse **K**inematics solver, meaning to **solve** the bones movement it goes _back and forward__ to solve it. There are other IK solvers in Godot but this is the one that worked better in my opinion.

__More information on [FABRIK3D](https://docs.godotengine.org/en/stable/classes/class_fabrik3d.html)__

##### `FABRIK3D` settings

```
Skeleton3D
└─  leg.0 (hip bone)
    └─  leg.1
        └─  leg.2
            └─  leg.3 (foot bone)
```

To create the spider's leg arch we need two items on the `Settings`, the first ones starting at the root bone `leg.0` and ending at the `leg.2` and using the **arch** (`Marker3D`) node as target.

This __arch__ node (it's the **pink sphere** on the video above) makes sure that the arch always __points up/outside__ and also makes more visually appealing/correct.

The last item on `Settings` applies to all the bones and uses the `FootMarker3D` as `Target Node` (it's the **blue ring** in the video above).

The `FootMarker3D` has `Top Level` as `true`, so it **does not** inherit the transformations of the parent, so it becomes __left behind__, not moving and rotating with the parent.


#### `RayCast3D`

The `RayCast3D` (seen in green) is used to know where the collision point with the scenario happened, and if the distance between the current position of the `FootMarker3D` to the collision point is too big then the **new target** position of the `FootMarker3D` should be the `Raycast3D` collision point.

The maximum distance between the collision position and `FootMarker3D`'s position uses the variable `step_max_distance` as it's value.

To make the movement of the `FootMarker3D` to the new position   more visually appealing this __relocation__ is done using a `Tween`, where the position gets moved like in a arch (lifting up midway to the new position).

The height of the tween's arch uses the variable `step_height` as it's value.
